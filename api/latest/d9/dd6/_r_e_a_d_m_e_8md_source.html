14 **not_null** is a 0-overhead modern utility for ensuring non-nullability
15 in a simple and coherent way.
16 
17 Unlike `gsl::not_null`, this type can work with move-only pointers like
18 `std::unique_ptr`, and does not require runtime checks unless explicitly
19 specified.
20 
21 *Stop worrying about nulls, and use `not_null` today!*
22

61 ## Background
62 
63 When working in a codebase, it's often difficult to know when any given pointer
64 may legally contain a null pointer value. This requires the user to do one of
65 two possible things:
66 
67 * Make an assumption that the pointer can never be `nullptr`, which risks the
68  posibility of a null-pointer dereference leading to undefined behavior, or
69 * Always check for `nullptr` before using -- resulting in more complicated code
70  paths, more branches, and potential pessimizations
71 
72 This is the driving force behind a `not_null` utility. The Guidelines Support
73 Library offers `gsl::not_null` to attempt to satisfy this, though this has
74 two notable limitations:
75 
76 * Construction and assignment of `not_null` is implicit, and requires runtime
77  checks unconditionally (which may result in a contract violation); for example:
78  ```cpp
79  // Performs check that the pointer is not null, when the expression can never
80  // take on the form of a non-null pointer
81  gsl::not_null<int*> p = new int{42};
82  ```
83 
84 * it does not work with move-only pointer types like `unique_ptr`, for example:
85  ```cpp
86  gsl::not_null<std::unique_ptr<int>> p = std::make_unique<int>(42);
87  // This errors due to 'p' not being movable!
88  gsl::not_null<std::unique_ptr<int>> q = std::move(p);
89  ```
90 
91 This is where this library comes in.
92 
93 Rather than allow implicit construction and assignment from the underlying
94 pointer, this library takes the hard stance of **correct on construction**,
95 which uses factory functions to produce the `not_null` objects. Two factories
96 are presented:
97 
98 * `check_not_null`, which checks a pointer for nullability before raising a
99  contract violation, which by default is thrown as an exception, or
100 * `assume_not_null`, which does no checking to produce the `not_null` and has
101  zero overhead, which is to be used for known non-null values
102 
103 With these, consumers are _required_ to consent to a check, or explicitly
104 state that their pointers are non-null, to reduce errors. This leaves any
105 remaining uses of `not_null` as nothing more than a 0-overhead wrapper; making
106 any runtime checks a manully consented operation. For example:
107 
108 ```cpp
109 // No check performed, since the expression can never be null!
110 cpp::not_null<int*> p = cpp::assume_not_null(new int{42});
111 
112 // Get pointer from API that we *expect* to be null, but cannot guarantee
113 auto q = cpp::check_not_null(get_pointer());
114 ```
115 
116 Additionally, `not_null` allows for move-constructors and move-assignment, since
117 reusing an object after a move is generally a logic-error, and there are many
118 pieces of tooling that will check for such a case. This allows better interop
119 with types like `unique_ptr`, and also allows for this `not_null` to communicate
120 with more legacy APIs by propagating such pointers out efficiently:
121 
122 ```cpp
123 // Should never be null, but not yet refactored to be 'not_null'
124 auto old_api(std::unique_ptr<Widget> p) -> void;
125 
126 auto new_api(cpp::not_null<std::unique_ptr<Widget>> p) -> void
127 {
128  // Extract the move-only unique_ptr, and push along to 'old_api'
129  old_api(std::move(p).as_nullable());
130 }
131 ```

133 ## Optional Features
134 
135 The defaults for this library should satisfy most consumers needs; however
136 **not_null** supports two optional features that may be controlled
137 through preprocessor symbols:
138 
139 1. Using a custom namespace, and
140 2. Disabling all exceptions
141 
142 ### Using a Custom Namespace
143 
144 The `namespace` that `not_null` is defined in is configurable. By default,
145 it is defined in `namespace cpp`; however this can be toggled by defining
146 the preprocessor token `NOT_NULL_NAMESPACE` to be the name of the desired
147 namespace.
148 
149 This could be done either through a `#define` preprocessor directive prior to
150 inclusion:
151 
152 ```cpp
153 #define NOT_NULL_NAMESPACE example
154 #include <not_null.hpp>
155 
156 auto test(example::not_null<int*> p) -> void;
157 ```
158 
159 <kbd>[Try Online](https://godbolt.org/z/1nd8Wj)</kbd>
160 
161 Or it could also be defined using a compiler flag with `-D` or `/D`, such
162 as:
163 
164 `g++ -std=c++11 -DNOT_NULL_NAMESPACE=example test.cpp`
165 
166 ```cpp
167 #include <not_null.hpp>
168 
169 auto test(example::not_null<int*> p) -> void;
170 ```
171 
172 <kbd>[Try Online](https://godbolt.org/z/x6rjej)</kbd>
173 
174 ### Disabling Exceptions
175 
176 By default, `not_null` will throw a `cpp::not_null_contract_violation` when
177 the contract has been violated (during factory construction).
178 
179 However, it is realized that not all developers appreciate or use exceptions,
180 and that environments that disable exception handling specifically may
181 error due to the existence of a `throw` expression. To account for this
182 possibility, exceptions may be disabled through the introduction of a simple
183 macro preprocessor symbol: `NOT_NULL_DISABLE_EXCEPTIONS`.
184 
185 In doing so, contract-violations will now behave differently:
186 
187 * Contract violations will call `std::abort`, causing immediate termination
188  (and often, core-dumps for diagnostic purposes)
189 * Contract violations will print directly to `stderr` to allow context for the
190  termination
191 * Since exceptions are disabled, there is no way to perform a proper stack
192  unwinding -- so destructors will _not be run_. There is simply no way to
193  allow for proper RAII cleanup without exceptions in this case.
194 
195 <kbd>[Try Online](https://godbolt.org/z/5na3KW)</kbd>

198 ## Compiler Compatibility
199 
200 **not_null** is compatible with any compiler capable of compiling valid
201 <kbd>C++11</kbd>. Specifically, this has been tested and is known to work
202 with:
203 
204 * GCC 5, 6, 7, 8, 9, 10
205 * Clang 3.5, 3.6, 3.7, 3.8, 3.9, 4, 5, 6, 7, 8, 9, 10, 11
206 * Apple Clang (Xcode) 10.3, 11.2, 11.3, 12.3
207 * Visual Studio 2015, 2017, 2019
208 
209 Latest patch level releases are assumed in the versions listed above.
